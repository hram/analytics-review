# 📌 КОНТЕКСТ
Постановка описывает backend/BFF сервис, который:
- агрегирует данные из нескольких downstream сервисов
- содержит fallback-логику
- должен быть реализован однозначно без трактовок

---

# 🎯 ЦЕЛЬ РЕВЬЮ

1. Найти ВСЕ проблемы постановки
2. Разложить их по категориям
3. Показать реальные последствия для разработки
4. Дать рекомендации:
   - как исправить конкретные проблемы
   - как улучшить сам формат постановок, чтобы такие ошибки больше не появлялись

# 🧩 СТРУКТУРА АНАЛИЗА

## 1. 🧠 ОБЩЕЕ ПОНИМАНИЕ (TL;DR)
- Кратко опиши, как ты понял логику системы
- Какие основные шаги выполнения запроса
- Какие сервисы участвуют
- Где критичные точки

---

## 2. 🔀 DATA FLOW И CONTROL FLOW

Опиши:
- полный pipeline выполнения запроса
- зависимости между вызовами
- где реально параллельность, а где последовательность
- какие данные переходят между шагами

Отдельно:
- построй логическую цепочку вида:
  INPUT → VALIDATION → SERVICE CALLS → AGGREGATION → RESPONSE

Выяви:
- разрывы потока
- неописанные переходы
- скрытые зависимости

---

## 3. 🌳 ВЕТВЛЕНИЯ И СЦЕНАРИИ (CRITICAL)

Составь ВСЕ возможные сценарии:

### Обязательные группы:
- happy path
- ошибки валидации
- отказ каждого downstream сервиса
- частичная деградация
- пустые ответы
- неожиданные ответы (например: null, пустые массивы, неконсистентные данные)

Для каждого сценария проверь:
- описан ли он в постановке
- однозначно ли поведение
- есть ли противоречия

Выдели:
❗ отсутствующие сценарии  
❗ неоднозначные сценарии  
❗ конфликтующие сценарии  

---

## 4. 🧾 КОНТРАКТ API (INPUT/OUTPUT)

Проверь:

### INPUT:
- все ли параметры описаны
- есть ли строгие правила валидации
- нет ли противоречий (например optional → required дальше)
- описаны ли edge cases

### OUTPUT:
- однозначно ли формируются поля
- различается ли:
  - null
  - 0
  - отсутствие поля
- есть ли конфликты источников данных

Выяви:
❗ неоднозначность контракта  
❗ риск разной реализации  

---

## 5. 🔌 DOWNSTREAM СЕРВИСЫ

Для каждого сервиса проверь:
- что считается SUCCESS
- что считается ERROR
- как обрабатываются:
  - HTTP ошибки
  - таймауты
  - пустые ответы
  - частичные данные

Выяви:
❗ отсутствующие правила обработки  
❗ разные трактовки success  

---

## 6. ⚠️ FALLBACK / DEGRADED MODE

Проверь:
- какие сервисы критичные, какие нет
- где используется fallback

Особое внимание:
- не подменяется ли "нет данных" → "валидные бизнес-значения"
- не ломает ли fallback бизнес-логику

Выяви:
❗ опасные fallback-решения  
❗ искажение данных  

---

## 7. 💣 КОНФЛИКТЫ ДАННЫХ

Проверь:
- есть ли несколько источников одного поля
- описано ли правило при конфликте

Например:
- hasPromo из разных сервисов
- price из разных источников

Выяви:
❗ неописанный приоритет  
❗ логические конфликты  

---

## 8. 🧮 БИЗНЕС-ЛОГИКА

Проверь:
- бизнес-правила явно описаны или “подразумеваются”
- есть ли скрытые assumptions

Выяви:
❗ implicit logic  
❗ недоописанные правила  

---

## 9. 🧯 ОБРАБОТКА ОШИБОК

Проверь:
- есть ли единая модель ошибок
- есть ли маппинг:
  service → BFF error
- нет ли дублей / конфликтов

Выяви:
❗ невалидную таблицу ошибок  
❗ несогласованность кодов  

---

## 10. ⏱ PERFORMANCE / TIMEOUT / PARALLELISM

Проверь:
- есть ли таймауты
- есть ли общий deadline
- корректно ли описана параллельность

Выяви:
❗ скрытые последовательные вызовы  
❗ риски latency  

---

## 11. 🧪 ТЕСТИРУЕМОСТЬ

Проверь:
- можно ли написать тесты по постановке
- хватает ли информации для:
  - unit
  - integration
  - contract tests

Выяви:
❗ невозможность тестирования  
❗ неоднозначные кейсы  

---

## 12. 🧱 РИСКИ РЕАЛИЗАЦИИ

Сформулируй:
- где разработчики начнут трактовать по-разному
- где появятся баги
- где будут “костыли”
